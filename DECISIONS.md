# DECISIONS

## Принятые решения

### 1. Формат проекта
Проект развивается как веб-каталог товаров fashion-сегмента.

### 2. Первый источник
Для первой рабочей реализации выбран `RINKAN`.

Это решение уже закреплено в артефактах проекта:
- `parser/rinkan_parser_v4.py`
- `rinkan_products_v4.json`
- `data/products.sample.json`

### 3. Текущий backend-стек
Для MVP выбран `FastAPI`.

Причины:
- быстрый запуск
- простой API для каталога
- удобно раздавать статические HTML-страницы

### 4. Текущий frontend-стек
Для MVP выбран простой статический frontend без фреймворка:
- `HTML`
- `CSS`
- `JavaScript`

Причины:
- быстрее собрать прототип
- меньше лишней инфраструктуры
- легче показать базовый каталог и страницу товара

### 5. Модель категорий
В актуальной схеме БД используется трехуровневая модель:
- `general_categories`
- `categories`
- `subcategories`

### 6. Идентификация товара
Товар в БД должен определяться по паре:
- `source_id`
- `source_product_id`

Это уже отражено в `sql/schema_v2.sql` через уникальное ограничение.

### 7. Промежуточный формат хранения
До реализации loader и PostgreSQL проект работает через sample JSON.

То есть текущий MVP осознанно использует промежуточную схему:
`parser -> JSON -> backend -> frontend`

### 8. Целевая следующая стадия
После MVP проект должен перейти на схему:
`parser -> loader -> PostgreSQL -> backend -> frontend`
